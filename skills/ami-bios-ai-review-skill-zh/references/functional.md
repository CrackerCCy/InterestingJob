# 功能分支

这个分支产出 `功能发现`。

重点关注变更是否会在目标启动阶段、平台配置、构建组成和用户可见固件行为上正确工作。

功能审查关注的是“行为正确性”和“集成正确性”，而不是编码风格。

## 输出规则

每条 finding 应说明：

- 文件类型或文件族
- 功能风险类别
- 缺陷或回归点
- 固件影响
- 联动复核目标

## 按文件类型的功能规则

### C / H / CPP

重点检查：

- 初始化顺序与阶段合法性
- Protocol / PPI 在使用点是否真的可用
- 寄存器编程顺序、必要延时、硬件状态假设
- 错误处理是否会悄悄改变平台行为
- Runtime、SMM、PEI、DXE、Recovery 路径正确性
- 对 NVRAM、CMOS、HOB、ACPI、handoff 数据的副作用

联动检查：

- `INF`、`DEPEX`、`DSC`、库绑定、Setup callback、ACPI 源

### ASM / S / NASM / ASM16 / CSM16 / INC / EQU / MAC / NASMB

重点检查：

- 入口与退出正确性
- 栈和寄存器状态
- 复位向量或早期初始化流程
- CPU 模式切换假设
- 与后续 C 代码阶段的衔接

联动检查：

- 入口声明、包装层、阶段文档、硅初始化模块

### INF / DEC / DSC / DEPEX

重点检查：

- 模块是否被正确纳入 / 排除
- 库解析是否正确
- PCD 值与特性开关是否符合预期平台行为
- 依赖表达式是否让模块启动过早、过晚或根本不启动
- 包声明是否与真实符号使用漂移

联动检查：

- 源码文件、`SDL`、`CIF`、平台装配

### SDL / CIF / MAK

重点检查：

- Token 默认值是否改变功能行为
- eLink 顺序与组件装配是否正确
- 板型 / 项目 / 变体选择是否漂移
- 构建变量是否与下游生成结果不再一致
- 元数据改动是否暗中改变集成行为

联动检查：

- `INF`、`DSC`、生成头文件、Setup / NVRAM 定义

### SD / HFR / VFR / UNI / SSP

重点检查：

- Setup 变量绑定是否正确
- suppress / gray-out / callback 逻辑
- 默认值和出厂值是否符合需求
- NVRAM / CMOS 字段布局是否与消费代码匹配
- UI 文本是否与真实功能一致
- 页面流转与导航一致性

联动检查：

- Setup 结构、callback 代码、`UNI`、`SSP`、策略模块

### ASL / ASLC / DXS

重点检查：

- ACPI 对象正确性与命名空间一致性
- Method 语义、返回值、副作用
- 设备暴露与 OS 兼容性假设
- 电源状态与热插拔逻辑
- 依赖源是否与协议发布顺序一致

联动检查：

- 驱动代码、协议生产者、平台策略、生成 AML 的来源

### 脚本与工具

适用于：

- `PY`、`BAT`、`SH`、`CMD`、`GAWK`

重点检查：

- 输入 / 输出路径处理是否正确
- 产物生成是否可重复
- 失败处理是否保持构建正确性
- 外部工具版本假设
- 打包 / 签名步骤是否可能生成不完整镜像

联动检查：

- 生成镜像、清单、发布产物、信任资产

### 配置 / 数据表

适用于：

- `YAML`、`JSON`、`XML`、`MXML`、`INI`、`CNF`、`DEF`、`DAT`、`DT`、`ASI`、`MCB`、`CBIN`、`TXT`

重点检查：

- 默认值与平台映射是否正确
- 板型间约束是否一致
- 寄存器、GPIO、训练参数、微码、FIT 数据是否正确
- 纯数据改动是否暗中改变运行时行为
- 文本形式的 FDF / 预留区声明是否影响 Flash 布局

联动检查：

- 消费代码、构建元数据、Flash 组成、安全流程

### 二进制 / 镜像 / 信任资产

适用于：

- `FD`、`FV`、`FFS`、`EFI`、`BIN`、`ROM`、`RAW`、`LIB`、`DLL`、`EXE`、`SO`、`PDB`、`CER`、`PEM`、`KEY`、`ESL`、`SIGSTRUCT`

重点检查：

- 来源是否符合预期
- 目标板型 / 目标变体是否正确
- 放置与打包是否正确
- Debug / Release 是否混淆
- 是否替换了二进制但没有同步更新元数据或源码

联动检查：

- 构建脚本、清单、`DSC`、`SDL`、FDF 文本声明

## 功能默认倾向

- 阶段正确性和行级正确性同样重要
- 元数据改动在 AMI 固件中可能是高功能风险改动
- 优先指出“让 bug 变成真实问题的集成点”，而不只是局部症状
- 对于模块首次引入提交，如果代码和元数据都改了，通常要主动寻找一条代码逻辑问题和一条元数据 / 集成问题
- 当源码依赖协议、库或资源时，必须检查相应 `INF`、`DEPEX`、`SDL` 或宿主集成路径是否真的让这种依赖在运行时安全成立
