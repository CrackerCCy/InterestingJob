# 风格分支

这个分支产出 `风格发现`。

重点关注 AMI Aptio 固件工程中的工程质量、一致性、清晰度与可维护性。

不要把个人偏好当成阻塞项。
只有当风格问题降低理解效率、制造集成歧义或增加长期风险时，才应升级为 finding。

## 输出规则

每条 finding 应说明：

- 文件类型或文件族
- 风格 / 可维护性类别
- 明确的不一致点
- 为什么它在 AMI BIOS 树里重要
- 若可能存在元数据漂移，说明联动检查目标

## 按文件类型的风格规则

### C / H / CPP

重点检查：

- EDK II 与项目本地命名是否一致
- 在行为不直观时，是否有清晰的阶段感知注释
- 是否存在可避免的宏滥用或过深条件编译
- 函数边界是否掩盖初始化 / 策略逻辑
- 头文件卫生、重复声明、平台常量位置是否合理

联动检查：

- 邻近头文件、库类使用、模块描述文件

### ASM / S / NASM / ASM16 / CSM16 / INC / EQU / MAC / NASMB

重点检查：

- 标签命名和控制流可读性
- 特权切换、栈假设、scratch 寄存器相关注释
- 常量定义在 include / macro 文件间是否一致
- 汇编组织是否便于后续维护者安全推理

联动检查：

- C 包装层、复位入口文档、include 片段

### INF / DEC / DSC / DEPEX

重点检查：

- 描述文件是否可读、归属是否准确
- 模块、库、GUID、PPI、Protocol 命名是否一致
- 是否存在冗余或过期声明
- 描述文件内容是否已与真实实现脱节

联动检查：

- 源码树、`SDL`、`CIF`、平台纳入点

### SDL / CIF / MAK

重点检查：

- Token 名称与 Help 文本是否准确反映真实行为
- eLink、组件、flavor、生成目标是否组织清晰
- 项目 / 包 / 特性元数据中是否存在可避免重复
- 构建开关名称是否已与真实行为漂移

联动检查：

- `DSC`、`INF`、Setup 文本、生成头文件

### SD / HFR / VFR / UNI / SSP

重点检查：

- Setup 控件名、Prompt、Help 与可见行为是否一致
- 宏组织是否保持可读性
- NVRAM 字段命名是否清晰并与消费代码一致
- UI 文本是否准确、稳定、可本地化

联动检查：

- callback 代码、变量结构、`UNI`、`SSP`、`SDL`

### ASL / ASLC / DXS

重点检查：

- ACPI 对象命名与命名空间是否一致
- Method 结构是否与平台意图一致且可维护
- 依赖源是否清晰、不重复
- 生成源约定是否仍便于平台团队理解

联动检查：

- 设备命名、ACPI include 片段、协议生产者

### 脚本与工具

适用于：

- `PY`、`BAT`、`SH`、`CMD`、`GAWK`

重点检查：

- 流程是否可读，失败处理是否明确
- 路径构造与选项命名是否稳定
- 调用模式是否可重复
- 共享构建脚本中是否残留临时本地逻辑

联动检查：

- 生成产物、发布打包、签名说明

### 配置 / 数据表

适用于：

- `YAML`、`JSON`、`XML`、`MXML`、`INI`、`CNF`、`DEF`、`DAT`、`DT`、`ASI`、`MCB`、`CBIN`、`TXT`

重点检查：

- 命名是否清晰
- Key / field 是否一致
- 是否有重复或陈旧配置项
- 表组织是否符合所属子系统
- 文本声明的文件名 / 标签是否已与真实用途不匹配

联动检查：

- 消费代码、平台文档、生成产物

### 二进制 / 镜像 / 信任资产

适用于：

- `FD`、`FV`、`FFS`、`EFI`、`BIN`、`ROM`、`RAW`、`LIB`、`DLL`、`EXE`、`SO`、`PDB`、`CER`、`PEM`、`KEY`、`ESL`、`SIGSTRUCT`

重点检查：

- 命名与放置是否一致
- 归属是否可追踪
- Debug / Release 标签是否正确
- 打包约定是否有助于来源审计

联动检查：

- 脚本、清单、Flash 组成输入

## 风格默认倾向

- 优先与周边包保持一致，而不是追求“教科书式纯净”
- 当陈旧元数据会误导未来审查者或集成者时，应升级为 finding
- AMI 元数据可读性很重要，因为它经常直接控制纳入关系和行为
