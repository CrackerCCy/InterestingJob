# 聚合 / 裁决分支

这个阶段消费：

- `安全发现`
- `风格发现`
- `功能发现`

并产出：

- `最终审查报告`

## 目的

把三个分支的结果合并成一个高信噪比报告。

目标不是重复罗列所有观察，而是给出一组最小但正确、工程上可执行的结论。

## 聚合规则

### 去重

如果多个分支实际上指向同一个根因，就应合并。

例如：

- 一个错误的 `SDL` token 默认值，同时带来安全和功能风险，应合并成一条主 finding，并保留安全 / 功能双标签
- 一个误导性的 `UNI` 字符串，如果只影响阅读可归为风格；若会导致操作员误判，则可升级为功能问题

### 优先级决议

当不同分支意见冲突时，优先级顺序为：

1. 安全影响
2. 功能影响
3. 风格影响

不要让风格分支去削弱已经成立的安全或功能严重度。

### 严重度决议

基于真实固件影响选择最高可辩护严重度：

- `P0`：启动关键损坏、信任边界失守、灾难性 Flash / 安全风险
- `P1`：高概率回归、权限问题、重大集成缺陷
- `P2`：真实的正确性、兼容性、可维护性问题
- `P3`：低风险润色或清晰度问题

### 范围控制

丢弃低信号评论，例如：

- 纯主观命名喜好
- 同一根因的重复表述
- 不改变工程动作的装饰性意见

保留：

- 真实 bug
- 安全弱点
- 集成不匹配
- 会导致误构建、误配置的元数据漂移

### 大 Diff 覆盖规则

对于范围较大的提交或模块首次引入，不要因为“去重”过度而让整类审查面消失。

以下情况应保留为独立 finding：

- 运行时逻辑 bug 与元数据依赖 bug
- 协议时序问题与内存 / 资源管理问题
- `INF` 依赖漂移与 `SDL` eLink 接入风险

如果某条高优先级 finding 已存在，并不意味着其它变化面的中优先级问题都应该被吞掉。
只有在“修复动作本质相同”时才应合并。

## 最终报告格式

先输出 findings。

结构如下：

```text
Final Review Report

[P?] 简短标题
File type: ...
Risk category: ...
Problem: ...
AMI BIOS relevance: ...
Cross-check: ...

[P?] 简短标题
...

Residual risks / validation gaps:
- ...
```

## 无问题场景

如果去重后没有剩余可执行问题，输出：

```text
Final Review Report

No findings.

Residual risks / validation gaps:
- ...
```

Residual risks 应点出主要盲区，例如：

- 未独立验证二进制来源
- 未做真实硬件运行验证
- 未将 ACPI 源与生成的 AML 对比
- 未在目标平台上验证 Setup 行为

## 聚合默认倾向

- finding 必须可执行
- 固件上下文优先于泛化的软件工程措辞
- 优先一条精准根因，而不是多个症状式 finding
- 必要时提邻近文件类型，帮助避免误诊
- 对于大 diff，优先保留 2-5 条“修复动作不同”的具体 finding，而不是 1 条笼统总括
